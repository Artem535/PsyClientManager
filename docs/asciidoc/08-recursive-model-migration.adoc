== Recursive Model Migration Plan

=== Goal

Move event/client data access from page-owned imperative refreshes to a recursive model structure where parent models expose child data through stable model indexes. The target shape is a single navigable domain tree:

[source,text]
----
Clients
  Client
    Notes
    Events
      Event
        Payment
        Reminder
Calendar
  Day
    Events
Analytics
  Period
    Aggregates
----

This should reduce duplicated reload logic, make UI pages react to the same source of truth, and keep derived views such as analytics and client details synchronized after mutations.

=== Current Constraints

- `QClientModel` owns the client list only.
- `QTimelineModel` owns events for the selected day only.
- Detail pages request data directly from `Database` through `Application` wiring.
- Analytics runs database aggregate queries and redraws independent charts.
- Event/client links are saved as side effects after event persistence.

=== Target Design

- Introduce a domain repository layer above `Database`.
: It exposes typed commands and query snapshots, emits change notifications with affected entity ids, and keeps SQL/DuckDB details out of UI models.
- Add a root recursive model based on `QAbstractItemModel`.
: Root nodes group clients, calendar days, analytics periods, and settings-backed presentation values.
- Keep specialized proxy models for pages.
: Client list, timeline, notes, and analytics should consume filtered/sorted projections instead of owning separate reload policies.
- Represent relationships explicitly.
: Client-event links, payment status, reminders, and notes become child nodes or relation records with stable ids.

=== Migration Steps

1. Inventory current model roles and page data dependencies.
: Document all roles from `QClientModel`, `QTimelineModel`, and detail pages. Mark each value as source data, relation data, or derived data.

2. Introduce repository notifications.
: Add signals such as `clientChanged(id)`, `eventChanged(id)`, `dayChanged(date)`, and `analyticsInvalidated(period)`. Keep existing database methods unchanged at first.

3. Move write operations into repository commands.
: Replace direct page/application calls to `Database::add_*`, `update_*`, and `remove_*` with repository methods. Preserve current validation behavior and conflict checks.

4. Build the recursive read model behind a feature flag.
: Implement the root model and populate it from repository snapshots. Do not switch pages yet.

5. Port one low-risk consumer first.
: Move analytics to repository notifications and read-only projections. This validates invalidation without editing workflows.

6. Port client list and client detail pages.
: Replace direct reloads with proxy models rooted at the recursive model. Verify selection stability after edits and deletes.

7. Port timeline and event dialog.
: The timeline should observe day nodes and event child nodes. Event creation and editing should call repository commands, then rely on model notifications for UI updates.

8. Remove duplicated reload paths.
: Delete old direct refresh calls after all pages consume projections. Keep focused regression tests around create/edit/delete flows.

=== Test Plan

- Unit-test repository commands against an isolated test database.
- Add model tests for parent/child indexes, row counts, role values, and stable ids.
- Cover mutation notification cases: add event, edit event time, delete client, change event-client link, update payment status.
- Add UI-level smoke checks for client list, timeline day switch, event edit, and analytics refresh.

=== Rollout Risks

- Recursive models are sensitive to invalid parent indexes and stale persistent indexes.
- Bulk refreshes can cause unnecessary `modelReset` calls and lose UI selection.
- Event-client relation changes can invalidate multiple projections at once.
- Analytics aggregates must not block the UI thread as notifications become more frequent.

Mitigate these by keeping repository snapshots immutable per notification, using `beginInsertRows`/`beginRemoveRows` instead of broad resets, and porting pages incrementally.
